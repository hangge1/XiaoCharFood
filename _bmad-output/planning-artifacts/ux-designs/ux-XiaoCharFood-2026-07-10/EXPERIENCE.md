# XiaoCharFood - Experience Spine

## Foundation

单一微信小程序移动端体验。MVP 不做桌面端、H5 独立站或多端同步体验。视觉身份参考 `DESIGN.md`，体验契约以本文件为准；两者冲突时，行为以 EXPERIENCE.md 为准，视觉以 DESIGN.md 为准。

[ASSUMPTION: MVP 采用微信小程序原生导航和底部 tab bar，不自建复杂导航框架。]

核心体验原则：

- 先让用户能轻松记录，再谈分析。
- 做饭建议必须解释“为什么推荐”，否则用户会把它当随机菜谱。
- 餐厅记忆用于私人找回，不引导公开评价。
- 当前版本不承诺多人共享，所有微文案避免“我们家一起编辑”等暗示。

## Information Architecture

| Surface | Reached from | Purpose |
|---|---|---|
| 首页 | 小程序打开、底部 tab | 今日三餐状态、快速记一餐、今晚吃什么入口 |
| 记一餐 | 首页主按钮、做饭计划转记录 | 创建或编辑饮食记录 |
| 周回顾 | 首页卡片、底部 tab 二级入口 | 查看最近 7 天饮食概览和低压力建议 |
| 做饭 | 底部 tab、首页建议卡片 | 请求做饭建议、查看买菜清单、加入今晚计划 |
| 做饭详情 | 做饭建议卡片 | 查看推荐理由、食材、加入计划 |
| 餐厅 | 底部 tab | 查看、筛选和新增餐厅记忆 |
| 餐厅详情 | 餐厅行点击 | 查看餐厅体验记录、编辑复购意愿 |
| 我的 | 底部 tab | 偏好设置、隐私说明、数据设置 |

推荐底部 tab：**首页 / 做饭 / 餐厅 / 我的**。周回顾从首页进入，暂不占用一级 tab，避免第一版导航过重。

→ Composition reference: `mockups/key-screens.html`. Spine wins on conflict.

## Voice and Tone

微文案服务低压力饮食觉察。品牌语气在 `DESIGN.md`，这里定义交互文案边界。

| Do | Don't |
|---|---|
| “本周外食 3 次，可以考虑明天清淡一点。” | “你本周外食超标了。” |
| “记录越多，建议越贴近你。” | “数据不足，无法分析。” |
| “要把这顿标记为放纵餐吗？” | “这顿不健康。” |
| “最近吃过类似的，仍然想选吗？” | “重复饮食，不推荐。” |
| “这是你的私人记录。” | “分享到广场让大家看看。” |

## Component Patterns

| Component | Use | Behavioral rules |
|---|---|---|
| 今日状态条 | 首页 | 展示早餐/午餐/晚餐是否已记录；点击空餐别进入记一餐并预填餐别。 |
| 快速记录按钮 | 首页、空状态 | 进入记一餐；按钮文案固定为“记一餐”。 |
| 记录表单 | 记一餐 | 餐别、图片、标签、备注任一核心内容即可保存；不强制热量。 |
| 标签选择器 | 记一餐、做饭 | 支持多选；放纵餐标签视觉上是普通标签的强调态，不作为错误态。 |
| 周回顾卡片 | 首页、周回顾 | 首页只展示 1-2 条摘要；完整趋势在周回顾页面。 |
| 场景标签组 | 做饭 | “今晚”“省事”“清淡一点”“别重复”“用掉食材”；支持 0 个或多个选中。 |
| 做饭建议卡片 | 做饭 | 每张卡片必须展示菜名、耗时、推荐理由和主要食材；点击进入详情。 |
| 买菜清单 | 做饭详情 | 支持勾选，不要求登录额外权限。 |
| 餐厅筛选条 | 餐厅 | 支持复购意愿、人均、场景、关键词；筛选后提供清除入口。 |
| 餐厅记忆行 | 餐厅 | 点击进入详情；长按不绑定删除，删除只在详情页或编辑页。 |

## State Patterns

| State | Surface | Treatment |
|---|---|---|
| 首次打开 | 首页 | 展示空的今日状态、主按钮“记一餐”、辅助入口“先看看今晚吃什么”。 |
| 无饮食记录 | 周回顾 | “记录几顿后，这里会帮你回顾最近吃得怎么样。”并提供“记一餐”。 |
| 记录保存成功 | 记一餐 -> 首页 | Toast “已记录。”，首页对应餐别状态更新。 |
| 没有照片 | 记一餐 | 允许保存；图片位显示“可选”。 |
| 权限拒绝 | 记一餐/餐厅 | 拍照、相册、定位失败时提供手动输入路径，不阻塞保存。 |
| 做饭历史不足 | 做饭 | 展示默认建议，并说明“记录越多，建议越贴近你”。 |
| 推荐重复 | 做饭 | 标注“最近吃过”，不强制隐藏。 |
| 无餐厅记忆 | 餐厅 | “去过的店可以记在这里。”提供“新增餐厅”。 |
| 筛选无结果 | 餐厅 | “没有符合条件的餐厅。”提供“清除筛选”。 |
| 删除确认 | 记录/餐厅详情 | 二次确认；文案说明删除后会影响回顾或历史记录。 |

## Interaction Primitives

- 点击为主，长按不承载关键功能。
- 表单保存不要求每项完整，核心是降低记录门槛。
- 图片上传、定位、订阅消息都必须可跳过。
- tab 切换不丢失未保存表单，离开前提示保存或放弃。
- 做饭建议可刷新，但刷新入口不应比“加入计划”更突出。
- 禁止连续弹窗式提醒；订阅消息是否启用留到后续决策。

## Accessibility Floor

- 所有交互控件点击区域不小于 44px。
- 标签和按钮不能只依赖颜色表达选中态，必须有文字/状态变化。
- 图片必须有可替代文字上下文，如餐别、店名、备注。
- 错误和权限失败使用明确文本，不只显示图标。
- 页面阅读顺序遵循视觉顺序：标题 -> 主要状态 -> 主操作 -> 次级内容。
- 文案不使用会增加焦虑的评价词。

## Responsive & Platform

微信小程序移动端优先，单列布局。适配常见窄屏手机，关键按钮不得因长中文文本溢出。底部 tab 和浮动动作之间保持安全间距。

平台权限降级：

- 相机拒绝：允许从相册选择或无图保存。
- 相册拒绝：允许拍照或无图保存。
- 定位拒绝：餐厅位置改为手动输入。
- 通知/订阅消息：MVP 不默认请求。

## Key Flows

### Flow 1 - 小周晚饭后记一餐

1. 小周打开首页。
2. 今日状态显示早餐、午餐、晚餐，其中晚餐未记录。
3. 小周点击“记一餐”或晚餐空状态。
4. 记一餐页预填“晚餐”。
5. 小周上传照片，选择“外食”“偏油”“吃撑了”，备注“烧烤吃多了”。
6. 点击保存。
7. **Climax:** 首页晚餐状态变为已记录，并显示“本周外食 3 次”的轻量回顾入口。

失败路径：照片上传失败 -> 允许不带照片保存，提示“照片没传上来，文字记录也可以。”

### Flow 2 - 小陈下班前决定今晚做什么

1. 小陈进入做饭 tab。
2. 选择场景标签“今晚”“省事”“清淡一点”“别重复”。
3. 点击“生成建议”。
4. 系统返回 3 张做饭建议卡片，每张都展示推荐理由。
5. 小陈打开“番茄牛肉豆腐汤”详情，查看主要食材。
6. 点击“加入今晚计划”。
7. **Climax:** 做饭详情展示买菜清单，小陈可以直接照着买。

失败路径：历史记录不足 -> 系统仍返回默认建议，并提示“记录越多，建议越贴近你。”

### Flow 3 - 周末找回上次吃过的餐厅

1. 小周进入餐厅 tab。
2. 选择“还想再去”“约会”“人均 100-150”筛选。
3. 列表缩小到几家店。
4. 点击餐厅行进入详情。
5. 查看上次菜品、照片、人均和备注。
6. **Climax:** 复购意愿和历史备注帮助两人快速决定是否再去。

失败路径：筛选无结果 -> 显示清除筛选入口，而不是要求新增记录。

## Open UX Questions

1. 放纵餐是普通标签中的一个，还是记一餐页的独立开关。
2. 首页第一屏是否优先展示今日饮食状态，还是“今晚吃什么”建议。
3. 做饭建议是否需要“刷新一批”，以及刷新次数是否要限制。
4. 餐厅新增是否需要地图/定位卡片，还是 MVP 仅手动输入。
5. 是否需要在“我的”中提供导出或清空数据。
